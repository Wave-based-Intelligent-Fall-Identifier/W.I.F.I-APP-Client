# W.I.F.Y 앱 바이브 코딩 회고 (짧은 버전)

> 작성: 2026-06-15 · 대상: ESP32(BLE) ↔ Android 앱 연동 + 앱 완성 작업

## 무엇을 했나
- **펌웨어 코드는 건드리지 않고**, 펌웨어 동작에 앱을 맞췄다. 펌웨어↔앱 통신 문서 3종 작성
  (`WIFY_BLE_PROTOCOL.md`, `WIFY_FIRMWARE_APP_DATA_EXCHANGE.md`, `DEBUG_INJECTION_SERVER.md`).
- 앱의 빠진 3대 기능 완성: **시스템 알림 / 데이터 영속성(JSON) / 백그라운드 포그라운드 서비스**.
- "초기화" 버튼을 실제 의미인 **베이스라인 재측정**으로 정정 + 펌웨어 신호 기반 **"측정 중…"** 표시.
- 실제 ESP32 없이 **Postman으로 가상 펌웨어 데이터를 주입**하는 디버그 HTTP 서버를 넣어 에뮬레이터에서 end-to-end 시연.

## 잘된 점
- **펌웨어가 단일 진실원**이라는 원칙을 끝까지 지켰다. 앱은 기기 상태를 미러링만 하고 임의로 만들지 않음.
- **"스냅샷 vs 에지" 계약**(Status=상태 미러·로그 없음 / Fall Event=로그·알림)을 세워, 중복 로그·10초 알림 스팸을 구조적으로 차단.
- **추상화 분리**(`BleController`/`WifyNotifier`/`WifyRepository` 인터페이스)로 실제·Fake·디버그 주입이 같은 경로를 공유 → 테스트가 쉬웠다.
- 외부 의존성 0(JSON 영속·HTTP 서버 모두 표준 라이브러리)으로 빌드가 안정적.

## 아쉬웠던 점 / 배운 것
- 처음엔 "측정 중"을 앱 타이머로 추정하려 했으나, **펌웨어 신호(flags bit3)로 받아야** 정확하다는 지적을 받고 바로잡음 → "기기가 진실원" 원칙을 다시 확인.
- 에뮬레이터 자동 탭(adb input)이 불안정해 일부 검증이 번거로웠다. 핵심 동작은 **영속 파일·logcat 직접 확인**으로 보완.
- 에뮬레이터 시스템 ANR이 자주 떠 노이즈가 됐지만, 앱 크래시와 구분해 판단.

## 펌웨어 팀에 남기는 핸드오프
1. 낙상 판정은 **ESP32 온디바이스 AI**. 결과만 기존 BLE로 보내면 앱은 그대로 동작(앱 변경 불필요).
2. CSI 수집 펌웨어와 BLE 펌웨어를 **한 칩에서 통합** 필요(`csi_data_calculate` printf 자리 → 추론 → `wify_notify_fall`).
3. 명령 `0x01` 수신 = **베이스라인 재측정**, 측정 동안 Status `flags.calibrating`(bit3) 1→0 notify.

## 현재 상태
- 앱: 스캔·연결·표시·알림·영속·백그라운드·재측정 완료. 화면 전수 클릭 점검에서 흰 화면/앱 크래시 0건.
- 남은 것은 **펌웨어 측 AI 낙상 판정 + 센싱↔BLE 통합** (앱 작업 아님).
